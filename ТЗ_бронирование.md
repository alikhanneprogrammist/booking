## 1. Общие сведения

### 1.1. Назначение системы

Система — собственный аналог Altegio: внутренний веб-сервис для **управления бронью объектов
комплекса OFFICE 2020** и ведения **базы клиентов**. Заменяет ручной учёт занятости и устраняет
двойные брони.

**Единица брони — объект целиком.** Бронируется **весь комплекс на 3 этажа** (караоке-зал,
сауна/хаммам/бассейн, банкетная зона, комнаты отдыха, кухня/еда — всё внутри одного объекта),
а не отдельная комната. Сауна, хаммам, караоке и зона еды — это **состав** объекта (для показа
и описания), а не отдельно бронируемые ресурсы.

По данным сайта office2020.kz бронируется **5 объектов** двух типов:
- **Комплексы** (3 этажа, целиком): «7 VIP», «2/5 VIP», «3/4 VIP».
- **Караоке-залы**: «6 VIP», «Банкетный».

### 1.2. Цели

1. Видеть **занятость всех зон в одном календаре**, 24/7, включая брони через полночь.
2. **Исключить двойные брони** одного зала на пересекающееся время (гарантированно, на уровне БД).
3. Ускорить оформление брони администратором смены (менеджер + 3 администратора).
4. Вести единую **базу клиентов** с историей визитов.
5. Заложить архитектуру под будущий публичный виджет записи, предоплаты и интеграции.

### 1.3. Границы MVP

**Входит в MVP:**
- Управление ресурсами и категориями.
- Календарь занятости + CRUD брони с защитой от пересечений.
- База клиентов (CRUD, поиск, история).
- Аутентификация сотрудников и роли (ADMIN / MANAGER).
- Двуязычный интерфейс RU / KK.

**НЕ входит в MVP** (закладывается архитектурно, реализуется позже):
- Публичный онлайн-виджет записи с сайта.
- Приём предоплат (Kaspi / TipTop Pay / карта).
- Авто-напоминания и воронки дожима (WhatsApp / Instagram).
- Интеграция с iiko / CRM.

## 4. Функциональные требования

### 4.1. Модуль «Аутентификация» (FR-AUTH)

- **FR-AUTH-1.** Вход по телефону + паролю. Пароль хранится только в виде bcrypt-хэша.
- **FR-AUTH-2.** Сессия на JWT; автоматический разлогин по истечении срока.
- **FR-AUTH-3.** Неавторизованный пользователь редиректится на `/login` (middleware).
- **FR-AUTH-4.** Деактивированный сотрудник (`isActive = false`) не может войти.
- **FR-AUTH-5.** Доступ к админ-разделам проверяется по роли на сервере (не только в UI).

### 4.2. Модуль «Объекты и доп.услуги» (FR-RES) — только ADMIN

- **FR-RES-1.** CRUD объектов: тип (`КОМПЛЕКС`/`КАРАОКЕ`), название (RU/KK), вместимость,
  цвет, фото (галерея), активность, порядок сортировки.
- **FR-RES-2.** Каждый объект хранит **состав**: список этажей с описанием и флаги удобств
  (караоке, финская сауна, хаммам, бассейн, банкетная зона, кол-во комнат отдыха, кухня).
  Отображается в карточке объекта и в диалоге брони.
- **FR-RES-3.** Каждый объект хранит **тарифы** (см. §4.9): цена за час, минимум часов (по умолч. 3),
  цена за полусутки (12 ч), цена за сутки (24 ч), фикс выходного дня, мин. депозит будней, примечание.
- **FR-RES-4.** CRUD справочника доп.услуг (бэк-вокал, танец живота, go-go, кальян, СПА,
  музоборудование): название (RU/KK), цена, единица (за вечер / за услугу), активность.
- **FR-RES-5.** Объект нельзя удалить при наличии связанных броней — только деактивировать;
  деактивированный объект скрыт из создания новых броней, но виден в истории.
- **FR-RES-6.** Фото объектов берутся из имеющихся материалов (`compress_images/photo/`) и/или
  с сайта office2020.kz и привязываются при первичном заполнении.

### 4.3. Модуль «Клиенты» (FR-CLI)

- **FR-CLI-1.** CRUD клиента: имя, телефон (уникальный), заметка, теги.
- **FR-CLI-2.** Поиск клиента по имени и телефону (по части строки).
- **FR-CLI-3.** Карточка клиента показывает историю всех его броней (по убыванию даты).
- **FR-CLI-4.** Создание клиента «на лету» прямо из диалога создания брони.
- **FR-CLI-5.** Телефон нормализуется к единому формату перед сохранением и проверкой уникальности.

### 4.4. Модуль «Календарь занятости» (FR-CAL)

- **FR-CAL-1.** Основной экран — таймлайн: объекты по колонкам (сгруппированы по типу:
  Комплексы / Караоке), время по вертикальной оси. Всего 5 колонок-объектов.
- **FR-CAL-2.** Поддержка режима **24/7**: сетка покрывает 00:00–24:00; брони, переходящие
  через полночь, отображаются корректно.
- **FR-CAL-3.** Переключение видов: **День** и **Неделя**; навигация вперёд/назад, кнопка «Сегодня».
- **FR-CAL-4.** Брони отображаются цветными блоками; цвет кодирует статус и/или категорию.
- **FR-CAL-5.** Клик по свободному слоту открывает диалог создания брони с предзаполненными
  ресурсом и временем.
- **FR-CAL-6.** Клик по существующей брони открывает диалог просмотра/редактирования.
- **FR-CAL-7.** Текущее время отображается индикатором (линия «сейчас»).
- **FR-CAL-8.** Все времена показываются в таймзоне `Asia/Almaty`.

### 4.5. Модуль «Бронирование» (FR-BOOK)

- **FR-BOOK-1.** Создание брони: объект, клиент, начало, конец, кол-во гостей, **тариф**,
  доп.услуги, статус, сумма, депозит/предоплата, комментарий, источник.
- **FR-BOOK-2.** При выборе объекта в диалоге показывается его **состав** (этажи, удобства) и
  доступные **тарифы** с ценами.
- **FR-BOOK-3.** **Авторасчёт стоимости** по §4.9: выбранный тариф × длительность (+ доп.услуги),
  с подстановкой нужной цены для будней/выходных. Итог редактируется вручную.
- **FR-BOOK-4.** Доп.услуги добавляются к брони из справочника (несколько, с количеством).
- **FR-BOOK-5.** Редактирование и **перенос** брони (смена времени и/или объекта).
- **FR-BOOK-6.** Отмена брони: статус `CANCELLED` (запись сохраняется, освобождает время).
- **FR-BOOK-7.** Статусы брони: `NEW`, `CONFIRMED`, `PREPAID`, `COMPLETED`, `CANCELLED`, `NO_SHOW`.
- **FR-BOOK-8.** Источник брони: `ADMIN`, `PHONE`, `WHATSAPP`, `INSTAGRAM`, `WIDGET` (последний — задел).
- **FR-BOOK-9.** Валидация: `конец > начало`; **минимальная длительность — 3 часа** для почасового
  тарифа (настраивается на объекте); кол-во гостей сверх вместимости объекта — предупреждение, не запрет.

### 4.6. Бизнес-правило: запрет двойных броней (FR-BOOK-OVERLAP) — **критично**

Реализуется **двумя независимыми барьерами**:

1. **Барьер уровня БД.** Exclusion constraint на таблице броней:
   ```sql
   CREATE EXTENSION IF NOT EXISTS btree_gist;
   ALTER TABLE "Booking" ADD CONSTRAINT booking_no_overlap
     EXCLUDE USING gist (
       "resourceId" WITH =,
       tstzrange("startAt", "endAt") WITH &&
     ) WHERE (status <> 'CANCELLED');
   ```
   Гарантирует на уровне базы, что две активные брони одного ресурса физически не пересекутся
   (защита от гонок при одновременной работе нескольких администраторов).

2. **Барьер уровня приложения.** Перед сохранением — проверка пересечений запросом
   (`resourceId = :id AND status <> 'CANCELLED' AND startAt < :newEnd AND endAt > :newStart`).
   При конфликте — понятная ошибка «Зал занят в это время» и подсветка конфликтующей брони в UI.
   Ошибка constraint из БД также перехватывается и маппится в это же сообщение.

> Граничное правило: интервалы полуоткрытые `[начало, конец)` — бронь, заканчивающаяся в 22:00,
> и бронь, начинающаяся в 22:00, **не** считаются пересечением.

### 4.7. Модуль «Сотрудники» (FR-USER) — только ADMIN

- **FR-USER-1.** CRUD сотрудников: имя, телефон, email (опц.), роль, активность.
- **FR-USER-2.** Назначение/смена роли (ADMIN / MANAGER).
- **FR-USER-3.** Сброс пароля сотрудника.
- **FR-USER-4.** Деактивация вместо удаления (сохранение истории действий).

### 4.8. Локализация (FR-I18N)

- **FR-I18N-1.** Полностью двуязычный интерфейс: русский и казахский.
- **FR-I18N-2.** Переключатель языка в шапке; выбор сохраняется.
- **FR-I18N-3.** Локализованные форматы даты/времени и чисел.
- **FR-I18N-4.** Справочные данные (объекты, доп.услуги) хранят названия на обоих языках (`nameRu`, `nameKk`).

### 4.9. Тарифы и расчёт стоимости (FR-PRICE)

Каждый объект имеет набор тарифов (по данным office2020.kz):

| Тариф | Описание | Применение |
|---|---|---|
| **Почасовой** | Цена за час, **минимум 3 часа** | Брони короче полусуток |
| **Полусутки (12 ч)** | Фиксированная цена за 12 часов | Объекты-комплексы |
| **Сутки (24 ч)** | Фиксированная цена за 24 часа | Объекты-комплексы |
| **Выходные (пт–сб)** | Фикс/минимум на выходной день | Влияет на цену/депозит |
| **Будни (вс–чт)** | Минимальный депозит по кухне/бару | Влияет на цену/депозит |

- **FR-PRICE-1.** Цены и минимум часов задаются на каждом объекте (FR-RES-3) и могут отличаться.
- **FR-PRICE-2.** Система определяет будни/выходные по дате брони (часовой пояс `Asia/Almaty`)
  и подставляет соответствующую цену/депозит.
- **FR-PRICE-3.** Авторасчёт даёт **предварительную** сумму; администратор может скорректировать
  итог и депозит вручную (бизнес-логика скидок/банкетов вариативна).
- **FR-PRICE-4.** Депозит/предоплата фиксируется в брони как сумма (приём онлайн-оплат — вне MVP).

> Эталонные цены для первичного наполнения — в таблице §5.8.

## 5. Модель данных

> Все временные поля — `timestamptz`, хранение в UTC, отображение в `Asia/Almaty`.

### 5.1. User (сотрудник)
`id`, `name`, `phone` (uniq), `email?`, `passwordHash`, `role` (`ADMIN`|`MANAGER`),
`isActive` (bool, default true), `createdAt`.

### 5.2. Resource (объект)
`id`, `kind` (`COMPLEX`|`KARAOKE`), `nameRu`, `nameKk`, `capacity` (int),
`color`, `photos` (string[]), `isActive` (bool), `sortOrder`.
**Состав:** `floors` (Json — список этажей с описанием), флаги удобств:
`hasKaraoke`, `hasFinnishSauna`, `hasHammam`, `hasPool`, `hasBanquet` (bool),
`restRooms` (int), `hasKitchen` (bool), `extraInfo?`.
**Тарифы:** `hourlyPrice` (decimal), `minHours` (int, default 3),
`halfDayPrice?` (12 ч), `fullDayPrice?` (24 ч), `weekendPrice?`,
`weekdayMinDeposit?`, `priceNote?`.

### 5.3. ServiceAddon (доп.услуга)
`id`, `nameRu`, `nameKk`, `price` (decimal), `unit` (`PER_EVENT`|`PER_ITEM`), `isActive` (bool), `sortOrder`.

### 5.4. Client (клиент)
`id`, `name`, `phone` (uniq), `note?`, `tags` (string[]), `createdAt`.

### 5.5. Booking (бронь)
`id`, `resourceId` → Resource, `clientId` → Client, `startAt`, `endAt`,
`status` (enum), `source` (enum), `tariff` (enum), `guests` (int),
`total` (decimal, default 0), `deposit` (decimal, default 0), `prepayment` (decimal, default 0),
`comment?`, `createdById` → User, `createdAt`, `updatedAt`.
**Ограничение:** exclusion constraint `booking_no_overlap` (см. FR-BOOK-OVERLAP).

### 5.6. BookingAddon (доп.услуга в брони)
`id`, `bookingId` → Booking, `addonId` → ServiceAddon, `qty` (int, default 1), `priceAtBooking` (decimal).

### 5.7. Перечисления
- `Role`: `ADMIN`, `MANAGER`.
- `ResourceKind`: `COMPLEX`, `KARAOKE`.
- `Tariff`: `HOURLY`, `HALF_DAY`, `FULL_DAY`, `WEEKEND`, `CUSTOM`.
- `BookingStatus`: `NEW`, `CONFIRMED`, `PREPAID`, `COMPLETED`, `CANCELLED`, `NO_SHOW`.
- `BookingSource`: `ADMIN`, `PHONE`, `WHATSAPP`, `INSTAGRAM`, `WIDGET`.

| NFR-3 | **Производительность.** Загрузка дневного календаря (все ресурсы) — < 1 с при типовом объёме. |
| NFR-4 | **Безопасность.** Хэширование паролей (bcrypt); серверная проверка ролей; защита роутов middleware; валидация входных данных (Zod). |
| NFR-5 | **Таймзона.** Хранение в UTC, отображение в `Asia/Almaty`. |
| NFR-6 | **Адаптивность.** Корректная работа на десктопе администратора; базовая адаптивность под планшет. |
| NFR-7 | **Локализация.** Без «зашитых» строк в коде — все тексты через next-intl. |
| NFR-8 | **Расширяемость.** Поля `source`, `prepayment` и enum `WIDGET` заложены под будущие виджет/оплаты. |

## 9. Критерии приёмки

Система считается принятой, если выполнены все проверки:

1. **Вход.** Демо-ADMIN успешно входит; неверный пароль и деактивированный аккаунт — отклоняются.
2. **Календарь / 24/7.** Создаётся бронь объекта «7 VIP» на 22:00–02:00 (через полночь) и корректно
   отображается на таймлайне.
3. **Анти-овербукинг (приложение).** Попытка создать вторую бронь того же объекта на 23:00–00:30
   отклоняется с сообщением «Объект занят».
4. **Анти-овербукинг (БД).** Прямой `INSERT` пересекающейся брони в БД отклоняется exclusion constraint.
5. **Граница интервалов.** Бронь 20:00–22:00 и бронь 22:00–23:00 одного объекта создаются обе (не конфликт).
6. **Минимум часов.** Почасовая бронь короче 3 часов отклоняется с понятным сообщением.
7. **Состав и тариф.** В диалоге брони «7 VIP» показан состав (3 этажа, сауна/хаммам/бассейн/караоке);
   при выборе тарифа «Сутки» сумма авто-подставляется (300 000 ₸) и редактируется вручную.
8. **Доп.услуги.** К брони добавляется кальян и СПА из справочника; итог пересчитывается.
9. **Перенос.** Бронь переносится на свободный слот другого объекта без ошибок.
10. **Клиент.** Клиент создаётся из диалога брони; в его карточке видна история броней.
11. **Роли.** MANAGER не видит и не может вызвать разделы «Объекты» и «Сотрудники»; ADMIN — видит и управляет.
12. **Языки.** Переключение RU ↔ KK меняет интерфейс и форматы дат.
13. **Целостность.** Активный объект с бронями нельзя удалить — только деактивировать.

## 10. Развилки и предпосылки внедрения

- **База данных.** Требуется экземпляр PostgreSQL (локальный или облачный, напр. Neon/Supabase)
  с возможностью включить расширение `btree_gist`. На текущей машине Postgres/Docker не установлены —
  способ предоставления БД согласуется до старта этапа 1.
- **Платформа разработки.** Windows 11 + PowerShell; запуск приложения и run-skill — под Windows
  (npm/Next.js dev server), без Linux-контейнера.
- **Фискализация/оплаты/мессенджеры** — вне MVP, см. §1.3.
