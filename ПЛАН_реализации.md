# План реализации — OFFICE 2020 (аналог Altegio)

> Рабочий план к `ТЗ_бронирование.md`. Дизайн-язык берём из **cal.com / cal.diy**,
> доменную архитектуру — из ТЗ (таймлайн ресурсов 24/7, анти-овербукинг на уровне БД).
> Среда: **Ubuntu 22.04** (а не Windows, как в §10 ТЗ).

---

## 0. Решение блокеров (до этапа 1) — обязательно

Проверка среды показала: установлен только `git`. Нет Node, npm, Postgres, Docker.

| Блокер | Решение | Действие |
|---|---|---|
| **Нет Node.js** | Поставить Node 20 LTS (nvm или apt/nodesource) | `nvm install 20` либо nodesource-репозиторий |
| **Нет PostgreSQL + `btree_gist`** | Два варианта — выбрать с заказчиком | см. ниже |
| **§10 говорит Windows** | Фактически Linux — обновить допущение в ТЗ | запуск под Ubuntu, dev-server `npm run dev` |
| **Нет фото залов** | Папка `compress_images/photo/` не найдена | запросить фото или тянуть с office2020.kz |

**PostgreSQL — развилка (нужно решение заказчика):**
- **Вариант A — облако (рекомендую для скорости):** Neon или Supabase. Бесплатный тариф,
  `btree_gist` доступен из коробки, `DATABASE_URL` в `.env`. Ничего не ставим локально.
- **Вариант B — локально:** `apt install postgresql-16`, `CREATE EXTENSION btree_gist;`.
  Полный контроль, но нужна установка и обслуживание.

> **Критично:** весь анти-овербукинг (§4.6 ТЗ) держится на расширении `btree_gist`.
> Без него exclusion-constraint не создать. Подтверждаем доступность ДО этапа 2.

## 1. Что берём из cal.com (дизайн-система)

Из cal.com заимствуем **визуальный язык**, НЕ доменную логику (там модель «event-types /
availability одного человека» — нам не подходит, у нас таймлайн ресурсов).

**Палитра и стиль:**
- Нейтральная база: `white` / `zinc-50` фон, границы `zinc-200`, текст `zinc-900`.
- Тёмная тема (`dark:` в Tailwind) — как в cal.com, переключатель в шапке.
- Скругления `rounded-md`, мягкие тени `shadow-sm`, много воздуха (щедрые отступы).
- Шрифт **Inter** (как у cal.com).
- Primary-кнопка — почти чёрная (`zinc-900`), вторичная — outline.
- **Цвет брони** на таймлайне берём из `Resource.color` (поле уже в модели §5.2).
- **Бэйджи статусов** брони — приглушённый цветной фон + тёмный текст
  (NEW=серый, CONFIRMED=синий, PREPAID=фиолетовый, COMPLETED=зелёный,
  CANCELLED=красный/перечёркнут, NO_SHOW=янтарный).

**Компоненты-паттерны (повторяем стиль cal.com):**
- Layout с **боковой навигацией** (Календарь / Клиенты / Объекты / Сотрудники) + шапка
  (язык, профиль, выход).
- **Диалоги/модалки** с чистыми формами (label сверху, поле, hint снизу).
- Кнопки, инпуты, селекты, тогглы — в минималистичном стиле cal.com.

**Чего НЕ берём:** tRPC-монорепо, Turborepo, пакеты, интеграции Google/Zoom/HubSpot,
публичный self-booking flow. Стек ТЗ (Next.js + Prisma + Tailwind + next-intl) и так
совпадает с базой cal.com — отдельный монорепозиторий не нужен.

## 2. Этапы реализации (детально)

Нумерация соответствует §8 ТЗ. Для каждого этапа — задачи, файлы, критерий готовности.

### Этап 1 — Скаффолд проекта
- Установить Node 20, инициализировать `npx create-next-app` (App Router, TS, Tailwind, ESLint).
- Поставить зависимости: `prisma @prisma/client`, `next-intl`, `next-auth@beta`, `zod`,
  `bcryptjs`, `date-fns date-fns-tz`.
- Настроить Tailwind под дизайн-токены cal.com (палитра, Inter, dark mode `class`).
- Структура папок по §6 ТЗ.
- `.env.example` + `.env` (DATABASE_URL, AUTH_SECRET).
- **Готово:** `npm run dev` поднимает пустой каркас с layout-ом в стиле cal.com.

### Этап 2 — Схема данных + анти-овербукинг + seed
- `prisma/schema.prisma`: модели User, Resource (+ состав + тарифы), ServiceAddon,
  Client, Booking, BookingAddon, все enum'ы (§5 ТЗ).
- Миграция; **отдельная raw-SQL миграция**:
  `CREATE EXTENSION btree_gist;` + exclusion-constraint `booking_no_overlap` (§4.6).
- `prisma/seed.ts`: реальные 5 объектов с тарифами из таблицы §5.8, доп.услуги,
  1 ADMIN + 3 MANAGER (демо-пароли через bcrypt), 2-3 клиента, демо-брони.
- **Готово:** БД заполнена; прямой INSERT пересекающейся брони отбивается БД.

### Этап 3 — Аутентификация и роли
- Auth.js (Credentials): вход по телефону + паролю, bcrypt-сверка.
- JWT-сессия, `middleware.ts` — редирект на `/login`, защита `(app)` роутов.
- Блокировка входа для `isActive=false`.
- Серверная проверка роли ADMIN для админ-разделов (не только UI).
- **Готово:** демо-ADMIN входит; неверный пароль и деактивированный — отбиваются.

### Этап 4 — Доменный слой брони (`lib/bookings.ts`)
- CRUD брони в транзакциях.
- Проверка пересечений на уровне приложения (барьер №2) + маппинг ошибки БД-constraint
  в понятное «Объект занят в это время».
- Валидация Zod: `endAt > startAt`, минимум 3 часа для HOURLY (настраивается на объекте),
  гости сверх вместимости — предупреждение, не запрет.
- **Авторасчёт стоимости** (§4.9): тариф × длительность, будни/выходные по `Asia/Almaty`,
  + доп.услуги; итог редактируется вручную.
- `lib/time.ts` — хелперы таймзоны (UTC ↔ Asia/Almaty), брони через полночь.
- **Готово:** серверная бизнес-логика покрыта; крайние случаи из §9 проходят.

### Этап 5 — Календарь занятости (главный экран)
- `components/calendar/ResourceTimeline.tsx`: 5 объектов в колонках
  (группы Комплексы / Караоке), время по вертикали, сетка 30-мин слотов, режим 24/7.
- `BookingBlock.tsx`: цветной блок (цвет ресурса), бэйдж статуса, брони через полночь.
- Виды **День / Неделя**, навигация ←/→, кнопка «Сегодня», линия «сейчас».
- Клик по пустому слоту → диалог создания (предзаполнены объект+время);
  клик по брони → просмотр/редактирование.
- `BookingDialog.tsx`: форма брони в стиле cal.com — показывает **состав** объекта
  (этажи, удобства), доступные **тарифы** с ценами, доп.услуги, авторасчёт.
- **Готово:** бронь «7 VIP» 22:00→02:00 (через полночь) отображается корректно;
  вторая пересекающаяся бронь отбивается.

### Этап 6 — Клиенты
- Список + поиск по имени/телефону (по части строки).
- Карточка клиента: история всех броней (по убыванию даты).
- `ClientPicker.tsx` — инлайн-создание клиента прямо из диалога брони.
- Нормализация телефона перед сохранением/проверкой уникальности.
- **Готово:** клиент создаётся из диалога брони; в карточке видна история.

### Этап 7 — Админка: объекты/доп.услуги + сотрудники/роли (только ADMIN)
- CRUD объектов (тип, названия RU/KK, вместимость, цвет, фото, состав, тарифы, сортировка).
- CRUD справочника доп.услуг.
- CRUD сотрудников: роль, активность, сброс пароля, деактивация вместо удаления.
- Запрет удаления объекта при наличии броней — только деактивация.
- **Готово:** MANAGER не видит/не вызывает админ-разделы; ADMIN управляет.

### Этап 8 — Локализация RU / KK
- `messages/ru.json`, `messages/kk.json`; все строки через next-intl (без хардкода).
- Переключатель языка в шапке, сохранение выбора.
- Локализованные форматы дат/времени/чисел.
- Справочники хранят `nameRu` / `nameKk`.
- **Готово:** переключение RU ↔ KK меняет интерфейс и форматы.

### Этап 9 — Приёмка + run-skill
- Прогон всех 13 критериев §9 ТЗ.
- README запуска, run-skill для локального старта.
- **Готово:** MVP принят по чек-листу.

## 3. Чек-лист приёмки (из §9 ТЗ — держим как цель)

1. Вход ADMIN / отказ при неверном пароле и деактивации.
2. Бронь через полночь 22:00→02:00 на таймлайне.
3. Анти-овербукинг (приложение) — отказ «Объект занят».
4. Анти-овербукинг (БД) — exclusion constraint отбивает INSERT.
5. Граница интервалов 20–22 и 22–23 — обе создаются.
6. Минимум 3 часа — короткая бронь отбивается.
7. Состав + тариф «Сутки» → авто 300 000 ₸, правится вручную.
8. Доп.услуги (кальян + СПА) — итог пересчитывается.
9. Перенос брони на другой объект.
10. Клиент из диалога + история в карточке.
11. Роли: MANAGER без админ-разделов.
12. Языки RU ↔ KK.
13. Целостность: активный объект с бронями не удаляется.
